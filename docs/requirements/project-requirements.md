# DailyHub - Project Requirements

## Overview

DailyHub 为 OpenClaw 机器人提供数字生活自动化任务策略，涵盖多平台签到、状态管理和反向控制三大场景。

---

## Core Requirements

### FR-001: 自动签到 (Automated Check-ins)

**描述**: 集成各平台的自动签到接口，每日自动执行。

**目标平台**:
- [x] 小桔充电（已落地）

**验收标准**:
- 每个平台可独立触发签到
- 签到结果（成功/失败/已签到）有明确返回
- 失败时有重试或告警机制

---

### FR-002: 全方位状态管理 (State Management)

**描述**: 集中监控各平台收益、积分余额及过期提醒。

**功能点**:
- [ ] 积分/余额查询
- [ ] 积分过期提醒
- [ ] 各任务完成状态汇总

**验收标准**:
- 支持单平台和全平台聚合查询
- 过期数据有时间维度提示
- 状态数据可结构化输出

---

### FR-003: 反向控制与退订 (Reverse Control)

**描述**: 提供退订营销短信、取消冗余订阅的能力。

**功能点**:
- [ ] 营销短信退订
- [ ] App 内自动订阅取消

**验收标准**:
- 退订操作可追踪（记录执行时间和结果）
- 误操作有确认机制

---

### FR-004: Skill 输出可读性约束 (Human-Readable Output)

**描述**: 每个可调用 skill 必须以“人可读结果汇报”为主，优先说明任务是否完成、做了什么、为什么失败。

**功能点**:
- [x] 每个 `SKILL.md` 包含 `Output Format` 章节
- [x] 每个 skill 在 `Output Format` 中定义自己的回复格式（可按场景不同）
- [x] 失败场景必须给出失败原因与下一步建议
- [x] 若附加结构化字段，仅作为补充，不替代人类可读汇报

**验收标准**:
- 所有现存 skill 文档均可检索到 `Output Format` 章节
- 每个 skill 的 `Output Format` 都有明确成功/失败汇报方式
- 失败场景包含“失败原因 + 建议动作”

---

## Non-Functional Requirements

| NFR | 描述 |
|-----|------|
| NFR-001 安全性 | API 密钥/Token 不得硬编码，通过环境变量或密钥管理服务注入 |
| NFR-002 可靠性 | 网络异常时有重试机制，重试次数可配置 |
| NFR-003 可观测性 | 每个任务执行有日志记录（时间、结果、耗时） |
| NFR-004 可扩展性 | 新平台集成遵循统一接口规范，不影响已有功能 |

---

## Out of Scope (当前版本)

- UI 界面
- 多用户支持
- 数据持久化存储（数据库）
